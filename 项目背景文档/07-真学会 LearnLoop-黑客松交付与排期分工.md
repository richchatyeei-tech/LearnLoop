## 1. 交付目标（6.30 必须达成）

### 1.1 一句话交付标准

> 交付一个**可交互**的双端 Web Demo：从「老师发布作业 → 学生提交作答 → AI 批改归因 → 老师确认讲评 → 学生订正闯关 → 掌握结果回流老师端」完整跑通；并配套黑客松评审材料（PPT + 演示脚本 + 项目说明）。

### 1.2 评委应看到的核心证明

| 证明点 | 对应能力 |
| --- | --- |
| 不是单点 AI 批改 | 批改后自动生成讲评、订正、回流 |
| 不是单端工具 | 同一份作业数据在教师 PC 与学生 Pad 间流动 |
| Agent 多步执行 | 批改 → 诊断 → 讲评建议 → 错因讲解 → 变式 → 掌握判定 |
| 人机协同 | 老师确认后才分发订正（交接点 A） |
| 品类差异 | 学生掌握结果回流改变老师决策（交接点 B） |

## 4. 技术架构与分工边界

### 4.1 推荐架构

```text
┌─────────────────────────────────────────┐
│  前端：现有原型 index.html + app.js      │
│  教师 PC Web + 学生 Pad Web               │
└──────────────────┬──────────────────────┘
                   │ REST API
┌──────────────────▼──────────────────────┐
│  后端：LearnLoop Engine                  │
│  · 状态机（Assignment.status）           │
│  · 事件：confirm_review / submit_remediation │
│  · fan-out / fan-in                      │
│  · SQLite / JSON 文件存储（黑客松够用）   │
└──────────────────┬──────────────────────┘
                   │ HTTP / 内部调用
┌──────────────────▼──────────────────────┐
│  算法服务（Skill）                        │
│  · grade_and_attribute（批改归因）        │
│  · review_suggest（讲评建议）             │
│  · explain_error（错因讲解）              │
│  · generate_variant（变式题）             │
│  · parent_summary（家长摘要，P1）         │
└─────────────────────────────────────────┘
```

### 4.2 角色分工

| 角色 | 核心职责 | 关键产出 | 不负责 |
| --- | --- | --- | --- |
| **产品 Owner（你）** | 控范围、验收 P0、演示脚本、答辩叙事、每日联调 | Demo 脚本、验收清单、PPT 大纲 | 不写复杂后端 |
| **产品队友** | Demo 数据、AI 输出样例、文案、PPT 用户故事/价值页 | 题目/作答/错因样例表、PPT 初稿 | 不扩 PRD 范围 |
| **后端** | 数据模型、API、状态机、fan-out/fan-in、seed 脚本 | 可启动服务 + API 文档 | 不做 OCR、不做复杂权限 |
| **算法** | 5 个 Skill 的 prompt + 结构化 JSON + fallback | 接口 + schema + 兜底数据 | 不做前端、不做状态机 |
| **前端/原型** | 现有原型接 API，保留 UI，加 loading/错误态 | 联调可点 Demo | 不重做视觉系统 |

> 若无专职前端：**Owner + 产品** 基于 `app.js` 改 fetch 调用；后端提供 CORS 与 mock 开关。

### 4.3 协作接口约定

**算法输出必须结构化**（参见 06 §2.1）：

```json
{
  "is_correct": false,
  "score": 0,
  "error_cause_category": "CAT-CONCEPT",
  "cause_description": "会根据公式计算，但未理解图像趋势与 k 正负的对应关系",
  "hit_kp_ids": ["KP-LF-03"]
}
```

**错因大类只允许**：`CAT-CONCEPT` | `CAT-READING` | `CAT-METHOD` | `CAT-CALC` | `CAT-EXPRESS`

**知识点只允许**：06 §1.3 九个 `kp_id`

## 5. API 最小集（后端负责定义，6.24 冻结）

| 方法 | 路径 | 说明 | 关联阶段 |
| --- | --- | --- | --- |
| GET | `/api/demo/bootstrap` | 初始化/重置 Demo 数据 | 全局 |
| POST | `/api/assignments` | 创建并发布作业 | ① 布置 |
| GET | `/api/assignments/:id` | 作业详情 + 状态 | 全局 |
| GET | `/api/assignments/:id/submissions` | 提交进度列表 | ③ 回收 |
| POST | `/api/assignments/:id/answers` | 学生提交作答 | ② 作答 |
| POST | `/api/assignments/:id/grade` | 触发 AI 批改 | ④ 批改 |
| GET | `/api/assignments/:id/grading` | 批改结果 + ErrorItem 列表 | ④ 批改 |
| POST | `/api/assignments/:id/grading/confirm` | 教师确认批改 | ⑤ 复核 |
| GET | `/api/assignments/:id/diagnosis` | 班级诊断 | ⑥ 讲评 |
| POST | `/api/assignments/:id/review/generate` | 生成讲评建议 | ⑥ 讲评 |
| POST | `/api/assignments/:id/review/confirm` | **交接点 A** 布置订正 | ⑦ |
| GET | `/api/students/:sid/tasks` | 学生订正任务列表 | ⑧ 闯关 |
| GET | `/api/students/:sid/tasks/:tid` | 单题闯关内容（讲解/变式） | ⑧ 闯关 |
| POST | `/api/students/:sid/tasks/:tid/submit` | 提交订正/变式结果 | ⑧ 闯关 |
| GET | `/api/assignments/:id/recap` | **交接点 B** 学情回收 | ⑨ 回收 |
| GET | `/api/assignments/:id/parent-summary` | 家长摘要（P1） | 沉淀 |

**状态流转**（与 PRD §9.1 对齐）：

```text
待作答 → 待批改 → 待确认 → 待讲评 → 订正回收中 → 已闭环 / 需二次干预
```

## 8. 五分钟的演示脚本（Owner 负责细化）

| 步骤 | 端 | 操作 | 讲解要点 |
| --- | --- | --- | --- |
| 1 | 教师 | 打开工作台 → 进入作业布置 → 发布「一次函数作业」 | 闭环从一次作业开始，不是孤立 AI 功能 |
| 2 | 学生 | 切换到 Pad → 做作业 → 提交 | 学生真实产生作答数据 |
| 3 | 教师 | 作业管理看提交 → 触发 AI 批改 | Agent 第一步：结构化批改 + 错因归因 |
| 4 | 教师 | 查看错因（概念性/审题等）→ 确认批改 | AI 先判，老师把关 |
| 5 | 教师 | 备课讲评 → 生成建议 → **确认布置订正** | **交接点 A**：1 份诊断 → N 个学生任务 |
| 6 | 学生 | 要弄懂的题 → 闯关（为什么错→换题试试→学会啦） | 不是给答案，是验证掌握 |
| 7 | 教师 | 切回学情回收 | **交接点 B**：掌握结果回流 |

## 11. 每日协作机制

### 11.1 站会（每天上午 15min）

每人仅报：

1. 昨日完成（对照里程碑）
2. 今日承诺交付（可验收）
3. 阻塞项（需要谁决策）

### 11.2 联调（每天晚 30min）

只问四个问题：

1. 批改前链路通了吗？
2. 交接点 A 写库了吗？
3. 学生闯关提交了吗？
4. 交接点 B 回流了吗？

### 11.3 决策原则

- **产品方向不再讨论**，只讨论 P0 实现细节
- **信息架构不砍，演示数据可砍**
- **两个交接点优先于一切 P2 可视化**

## 14. 阶段性结论

**6.30 的唯一成功标准**：

> 评委看到一份作业从发布到掌握回流的全链路真实交互，并理解「真学会 LearnLoop = 同一份数据 + Agent 编排 + 双端闭环」，而非又一个 AI 批改工具。

**团队行动优先级**：

1. 后端状态机 + 两个交接点  
2. 算法结构化输出 + fallback  
3. 前端接 API（保留现有 UI）  
4. 产品 Demo 数据 + 答辩材料  
5. 学情溯源等 P2 仅作静态展示
