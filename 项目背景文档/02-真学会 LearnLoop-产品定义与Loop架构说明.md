## 2. 产品定义

### 2.1 一句话定义

**真学会 LearnLoop** 是一款面向中小学作业与测验场景的 **作业后 AI 学习闭环产品**：基于同一份作业数据，为教师生成讲评与分层干预方案，为学生生成错因讲解、订正任务与针对性补练，并为家长提供可理解的协同摘要，让一次作业从「批完」真正走向「真学会」。

### 2.2 品类定义

| 项目 | 内容 |
| --- | --- |
| 中文名 | **真学会** |
| 英文名 | **LearnLoop** |
| 品类 | 作业后的 AI 学习闭环产品 |
| 主 Slogan | **批完不算完，真学会才算。** |
| 副 Slogan | 一份作业，老师知道怎么讲，学生知道怎么会，家长知道怎么配合。 |

### 2.3 与竞品的本质差异

| 产品类型 | 解决什么 | 数据流 | 终点 |
| --- | --- | --- | --- |
| 作业帮 / 拍题工具 | 作业怎么做、答案是什么 | 学生单向获取 | 知道答案 |
| 智慧作业 / 批改平台 | 作业怎么批、统计怎么看 | 批完 → 报表 | 批完 |
| 错题本 / 题库 | 错题怎么存、怎么刷 | 题目沉淀 | 做过更多题 |
| **真学会 LearnLoop** | 批完之后怎么**真学会** | 批完 → 行动 → 验证 → 回流 | **Loop 关闭（掌握）** |

核心差异一句话：

> 别人解决「怎么更快批完」，我们解决「批完之后怎么真学会」。

### 2.4 品牌与产品机制的关系

- **真学会**：三个 Loop 共同指向的 **关闭条件**（终点状态）。
- **LearnLoop**：让「真学会」可重复发生的 **产品机制**（过程系统）。

## 3. Loop：核心产品概念

### 3.1 Loop 是什么

在教育场景里，**Loop 不是流程图画个圈**，而是同时满足四件事的 **可重复、可验证、可迭代的闭环**：

| 要素 | 含义 | 作业场景举例 |
| --- | --- | --- |
| **输入** | 可观测的学习事实 | 作业、作答、错因、掌握状态 |
| **行动** | 基于事实生成的下一步 | 讲评、订正、变式补练、面批 |
| **验证** | 检验行动是否有效 | 变式对错、班级掌握率、二次错误 |
| **回流** | 验证结果改变下一轮决策 | 仍薄弱 → 二次干预；已掌握 → 关闭 Loop |

**Loop 没关上** = 做了动作但没有验证，或验证了但没有改变下一步。

现有产品大多断在 **验证** 与 **回流**：能批、能存、能练，但很少回答「练完有没有真会」「讲评有没有效」。

### 3.2 Loop 是产品概念，不是现成技术框架

Loop 与实现的关系，可类比电商中的「订单」：

| 层级 | Loop 是什么 | 举例 |
| --- | --- | --- |
| **产品概念** | 用户能感知的一轮「做完并验证」 | 「这次讲评有没有效」「这道题真会不会」 |
| **产品设计** | 状态、页面、进度、关闭条件 | 老师见「待验证 8 人」；学生见「待变式验证」 |
| **技术实现** | 数据 + 状态机 + 规则 + AI 调用 | 表字段存状态；事件驱动下一步 |

结论：

- Loop **首先是产品概念**，定义「什么叫一轮闭环、何时关闭」。
- 技术上用 **状态机 + 共享数据模型 + 事件触发** 即可实现，**不必**先造一个叫 LoopEngine 的复杂框架。
- Agent 是 Loop 中 **某些步骤的智能执行者**，不是 Loop 本身。

### 3.3 三个 Loop

真学会的产品设计核心，是把三个 Loop 设计清楚、跑通、彼此咬合：

```text
┌─────────────────────────────────────────────────────────┐
│                    Dual Loop（教与学咬合）                  │
│  同一份作业数据在老师侧、学生侧之间双向流动，直到 Loop 关闭   │
└─────────────────────────────────────────────────────────┘
         ▲                                    ▲
         │                                    │
┌────────┴────────┐                  ┌────────┴────────┐
│  Teaching Loop  │                  │  Learning Loop  │
│   （教学闭环）    │                  │   （学习闭环）    │
└─────────────────┘                  └─────────────────┘
```

#### （1）Teaching Loop — 教学闭环（老师侧）

**本质**：把「批完作业」变成可迭代的教学决策系统。

```text
诊断 → 决策 → 执行 → 验证 → 再诊断
```

| 阶段 | 老师要什么 | 常见断点 |
| --- | --- | --- |
| 诊断 | 班级错在哪、共性错因 | 靠经验翻卷子，耗时长 |
| 决策 | 讲什么、谁先讲、谁要跟进 | 只有正确率，缺少错因结构 |
| 执行 | 课堂讲评、分层任务、个别辅导 | 讲评准备与分发割裂 |
| 验证 | 讲评后学生是否真掌握 | **讲完即结束** |
| 再诊断 | 哪些题还要讲、谁还要追 | 很少基于订正/补练回流 |

**Loop 关闭条件（示例）**：班级目标错因掌握率达到阈值，或「需跟进名单」已处理完毕。  
**不是**「课上讲完了」。

#### （2）Learning Loop — 学习闭环（学生侧）

**本质**：把「订正」从抄答案，变成可验证的掌握过程。

```text
错因 → 理解 → 订正 → 验证 → 沉淀
```

| 阶段 | 学生要什么 | 常见断点 |
| --- | --- | --- |
| 错因 | 知道「我为什么错」 | 拍题直接给答案 |
| 理解 | 用自己懂的方式讲通 | 解析太标准、看不懂 |
| 订正 | 按步骤改，不是抄结果 | 订正=改个答案 |
| 验证 | 换题还会不会 | **订正完就结束** |
| 沉淀 | 掌握状态可追踪 | 错题本只存题 |

**Loop 关闭条件（示例）**：变式补练通过 → `已掌握`。  
**不是**「订正已提交」。

#### （3）Dual Loop — 教与学咬合（产品品类定义）

前两个 Loop 可各自做成独立产品；**Dual Loop 是真学会与作业帮、错题本、智慧作业的根本差异**：

```text
老师诊断 ──分发──→ 学生订正 / 补练
      ↑                    │
      └── 掌握反馈 ←────────┘
```

| 连接点 | 价值 |
| --- | --- |
| 诊断 → 任务 | 老师讲什么，学生练什么，同源同向 |
| 任务 → 反馈 | 老师看到的不是「交没交」，而是「会不会」 |
| 反馈 → 再诊断 | 讲评无效的题/人自动浮出 |
| 长期记忆 | 跨次作业看错因是否反复、讲评是否有效 |

家长端是 Dual Loop 的 **只读协同视图**：不参与关闭 Loop，但能看到进度与配合建议。

## 5. Agent：如何理解，以及如何落地

### 5.1 Agent 在本项目中的定义

**Agent ≠ 多个自主智能体互相委派。**

在本项目中，Agent 指：

> **围绕 Loop 自动执行多步任务的工作流**：在关键步骤调用大模型与工具，在关键节点保留教师确认，由规则与状态机保证流程可控。

与普通功能的区别：

| 普通功能 | Agent（工作流视角） |
| --- | --- |
| 固定输入 → 固定输出 | 理解上下文 → 多步执行 → 结构化输出 |
| 例：算正确率 | 例：错因聚类、讲评建议、订正引导、掌握判定 |

### 5.2 「三个 Agent」是能力分工，不是必须的 Multi-Agent 架构

讨论中常出现三个名称：

| 名称 | 产品含义 | 推荐技术映射 |
| --- | --- | --- |
| 教学 Agent | 帮老师完成诊断与讲评决策 | **Review Workflow**（讲评诊断工作流） |
| 学习 Agent | 帮学生完成订正与掌握验证 | **Mastery Workflow**（订正补练工作流） |
| 教与学 Agent | 连接老师与学生两侧 | **Loop 编排层**（状态机 + 事件，**不是**第三个 LLM Agent） |

**教与学「连接」的典型实现**就是普通业务逻辑：

```text
teacher_confirmed  →  create_student_loops()
student_completed  →  aggregate_and_notify_teacher()
```

### 5.3 为什么不推荐比赛阶段做 Agent-to-Agent

| 场景 | 是否需要 Multi-Agent |
| --- | --- |
| 固定流程：批改 → 讲评 → 订正 → 验证 → 反馈 | **不需要** |
| 老师确认后按规则分发 | **不需要** |
| 按错因生成讲解与变式 | **不需要**（链式 LLM 即可） |
| 路径高度动态、需多轮自主规划 | 可能需要 |
| Agent 之间自然语言协商 | 教育场景通常 **不需要** |

教育场景更需要：**可解释、可审计、教师可确认**。  
推荐：**Workflow 为主，LLM 为辅** —— 即 **Agentic Workflow**，而非 **Multi-Agent System**。

### 5.4 答辩口径（架构被质疑时）

> 我们说的不是三个独立自治的 Agent，而是一个 LearnLoop 引擎上的三类智能能力：教师讲评、学生学习、双端反馈。技术上是一个状态机加共享数据模型，在关键步骤调用大模型。AI 负责「诊断、讲解、出题」，规则和教师确认负责「何时分发、何时关闭 Loop」。这样既体现 Agent 多步执行，又保证教学流程可控。

## 6. 核心使用主线与功能模块拆分

本节把前面的抽象 Loop 概念，收敛为一条 **可落地的双端使用主线**，再从主线拆出功能模块，作为架构设计的直接依据。

### 6.1 主线：一份作业的接力赛

不要把产品理解为「老师端产品 + 学生端产品」，而应理解为 **一根接力棒，在老师与学生之间传递两次**。值得做的主线只有这一条，且刚好串起背景预研中的「场景一~四」用户故事：

```text
【场景零】老师布置作业 → 学生作答提交 → 作业管理查看进度
【场景一】AI 批改 + 错因诊断 → 老师确认批改 → 老师确认讲评方案
                                    │
                          ★交接点 A：老师 → 学生（fan-out 分发）
                                    │
【场景二】学生看错因讲解 → 订正 → 变式补练 → 掌握判定
                                    │
                          ★交接点 B：学生 → 老师（fan-in 汇总）
                                    │
【场景三】老师看掌握复盘 → 决定二次干预 / 关闭 Loop
                                    │
【场景四】家长看协同摘要（只读支线，不在主干上）
```

定位说明：

- 学生不连老师的单独使用，是这条主线的 **退化形态**，不是第二条主线。
- 家长是主线的 **只读支线**。
- 比赛阶段把这一条接力跑顺，优先级高于铺多条线。

### 6.2 主线的灵魂：两个交接点（fan-out / fan-in）

单端能力（批改、讲解、出题）到处都有；真正稀缺、也是「双端闭环」实体的，是两个交接点。

| | 交接点 A（老师→学生） | 交接点 B（学生→老师） |
| --- | --- | --- |
| 动作 | fan-out：1 份诊断 → N 个个人任务 | fan-in：N 个掌握结果 → 1 张复盘 |
| 价值 | 学生练的，正是老师诊断出的问题 | 老师看到的是「会没会」而非「交没交」 |
| 竞品现状 | 智慧作业止步于此（出报表，不生成个人任务） | 几乎无人做（学生练完不回流给老师） |

- 作业帮：只有学生侧，无交接点。
- 智慧作业：有 A 的雏形（能布置），无 B（掌握不回流改变教学）。
- 真学会：A、B 都做，**B 是杀手锏**。

**设计取舍**：工程资源优先投向两个交接点，而非把 OCR / 批改这类「人人都有」的部分做到极致。

### 6.3 贯穿主线的原子：ErrorItem 的六种形态

主线能跑通，是因为底下有 **一个原子数据对象** 在流动、变形 —— 错题项 `ErrorItem`：

```text
ErrorItem {
  题目, 知识点(kp_id),
  学生作答, 是否错,
  error_cause_category,   // 5 类封闭枚举，见 06 §2.2
  cause_description,      // AI 按题生成，老师可改
  掌握状态
}
```

同一个对象，在主线上经历六个生命周期阶段：

| 阶段 | 形态 | 所在场景 |
| --- | --- | --- |
| ① 生成 | 批改后产出，带知识点 + 错因类型 | 场景一（诊断） |
| ② 聚合 | 按错因/知识点聚类 → 班级错因分布 | 场景一（讲评决策） |
| ③ 分发 | 按学生 fan-out → 个人订正任务 | **交接点 A** |
| ④ 学习 | 挂上「错因讲解 + 订正 + 变式」 | 场景二 |
| ⑤ 判定 | 写回掌握状态（会 / 不会） | 场景二结尾 |
| ⑥ 沉淀 | 进入长期记忆，跨作业追踪 | 贯穿、场景三复盘 |

架构线索：**班级级是 ErrorItem 的聚合视图，学生级是分发视图，长期记忆是历史流水。一个对象，三种视图。**

### 6.4 从主线拆功能模块（9 个）

按主线顺序拆，并标注性质（基础设施 = 够用即可；护城河 = 重点投入）：

| # | 模块 | 主线位置 | 性质 | LLM / 规则 |
| --- | --- | --- | --- | --- |
| 1 | 作业接入与结构化 | 场景一入口 | 基础 | OCR + LLM |
| 2 | 批改与错因归因引擎 | 场景一 | 半护城河（错因质量是地基） | LLM + 规则 |
| 3 | 教师讲评决策台（含确认闸门） | 场景一 | 差异化 | LLM 出建议 + 人工确认 |
| 4 | **任务分发模块（交接点 A）** | A | **护城河** | 规则为主 |
| 5 | 学生学习模块（讲解/订正/变式/判定） | 场景二 | 差异化 | LLM + 规则判定 |
| 6 | **掌握汇总与回流模块（交接点 B）** | B | **核心护城河** | 规则为主 |
| 7 | 教师复盘与二次干预 | 场景三 | 差异化 | LLM + 规则 |
| 8 | 家长摘要 | 场景四（支线） | 锦上添花 | LLM |
| 9 | 学情记忆 / 沉淀 | 贯穿 | 长期价值 | 数据 + 检索 |

工程精力建议压在 **3、4、5、6**，尤其是交接点 4 与 6 —— 它们是「双端闭环」叙事的实体。

## 7. 产品架构分层（推荐）

### 7.1 总体结构（落地版四层）

```text
┌───────────────────────────────────────────────────────┐
│  编排引擎  LearnLoop Engine                             │
│  状态机 + 事件总线（fan-out / fan-in 都发生在这里）       │
└───────────────────────────────────────────────────────┘
        │  事件驱动                       ▲ 结果回写
        ▼                                │
┌──────────────────┐            ┌──────────────────┐
│  Review Workflow │            │ Mastery Workflow │
│  教师侧：诊断/讲评 │            │  学生侧：讲解/订正 │
│  /确认/复盘       │            │  /变式/判定       │
└──────────────────┘            └──────────────────┘
        │                                │
        └──────────────┬─────────────────┘
                       ▼
┌───────────────────────────────────────────────────────┐
│  Skill 原子能力（无状态）                                │
│  OCR · 批改 · 错因归因 · 讲评生成 · 错因讲解 ·           │
│  变式生成 · 掌握判定 · 摘要生成                          │
└───────────────────────────────────────────────────────┘
                       │ 读写
                       ▼
┌───────────────────────────────────────────────────────┐
│  共享数据底座（接力棒）                                  │
│  ErrorItem（原子）· 知识点体系 ·                         │
│  HomeworkLoop（班级级）· StudentLoop（学生级）· 学情记忆 │
└───────────────────────────────────────────────────────┘
```

要点：两条工作流 **互不调用**，全靠编排引擎的事件 + 共享数据「隔空传棒」；交接点 A 前有「老师确认闸门」，AI 只出建议，是否分发由规则 + 人决定。

### 7.2 四层职责

| 层级 | 职责 | 技术形态 |
| --- | --- | --- |
| **编排引擎** | 唯一负责「交接」：状态流转、fan-out、fan-in | 状态机 + 事件总线 |
| **双工作流** | 教师侧 / 学生侧多步任务编排，彼此不直接通信 | 工作流（代码或 Dify / LangGraph 等） |
| **Skill 原子能力** | 无状态，输入→输出，可单测、可换模型 | LLM API + 工具调用 + 规则 |
| **共享数据底座** | 全系统唯一事实来源，三端共读共写 | 数据库（ErrorItem / Loop 表 / 记忆） |

### 7.3 核心对象模型（概念层）

```text
LoopInstance（一次作业的闭环实例）
├── 作业数据（题目、作答、批改、错因）
├── TeachingLoopState（老师侧状态与动作）
├── StudentLoop[]（每个学生的子 Loop）
├── DualLoopState（双端咬合：分发、汇总、二次干预）
└── LoopOutcome（关闭结果：掌握率、待跟进名单、家长摘要）
```

**StudentLoop** 内部阶段：

```text
错题 → 错因 → 订正 → 变式 → 掌握状态（已掌握 / 仍薄弱 / 需升级干预）
```

**一份作业 = 一个 LoopInstance**；教师端、学生端、家长端是同一实例的不同视图，共享错因结构与掌握状态。

### 7.4 核心数据表（最小集）

#### HomeworkLoop（班级级）

| 字段 | 说明 |
| --- | --- |
| id | Loop 实例 ID |
| homework_id | 关联作业 |
| status | 见下表「用户语状态」（与 PRD §9.1 一致） |
| teacher_confirmed_at | 老师确认讲评方案时间 |
| class_mastery_summary | 班级掌握汇总（验证后） |

**status 用户语 ↔ 内部实现（研发用）**：

| 用户语（PRD / UI） | 内部实现参考 |
| --- | --- |
| 待作答 | `pending_answer` |
| 待批改 | `pending_grade` |
| 待确认 | `pending_confirm` |
| 待讲评 | `pending_review` |
| 订正回收中 | `remediating` |
| 已闭环 | `closed` |
| 需二次干预 | `needs_intervention` |

> 界面与 PRD **只使用用户语**；内部英文键仅供后端状态机，见 [00 文档索引](./00-真学会%20LearnLoop-文档索引.md)。

#### StudentLoop（学生级）

| 字段 | 说明 |
| --- | --- |
| id | 学生 Loop ID |
| homework_loop_id | 关联班级 Loop |
| student_id | 学生 |
| mastery_status | `pending` / `mastered` / `still_weak` |
| error_items[] | 错题、错因、订正、变式、判定结果 |

### 7.5 两个交接点的事件实现

「教与学」的连接，不是第三个 Agent，而是编排引擎里的两段事件处理：

```text
on(teacher_confirmed_review):        // 交接点 A：fan-out
    读 HomeworkLoop 的班级错因
    → 按学生拆分生成 StudentLoop[]
    → 触发 Mastery Workflow

on(all_students_verified):           // 交接点 B：fan-in
    读所有 StudentLoop 的掌握状态
    → 聚合成班级复盘
    → 写回 HomeworkLoop
    → 触发 Review Workflow（复盘 / 二次干预）
```

### 7.6 关键事件与流转

| 事件 | 触发条件 | 系统动作 | Assignment.status → |
| --- | --- | --- | --- |
| `assignment_published` | 老师发布作业 | 写 Assignment + AssignmentItem | 待作答 |
| `answers_submitted` | 学生提交齐 / 到截止 | 写 Answer | 待批改 |
| `grading_completed` | AI 判分 + 错因归因完成 | 写 Judgement / ErrorItem | 待确认 |
| `grading_confirmed` | 老师确认批改 | 写 teacher_confirmed | 待讲评 |
| `teacher_confirmed_review` | 老师确认布置订正（交接点 A） | fan-out → StudentLoop / Remediation | 订正回收中 |
| `student_practice_submitted` | 学生提交变式 | 掌握判定 → 更新 ErrorItem | 订正回收中 |
| `verification_completed` | 汇总完成（交接点 B） | fan-in → 复盘 / 家长摘要 | 已闭环 / 需二次干预 |

### 7.7 LLM 与规则的分工

| 适合 LLM | 适合规则 / 代码 |
| --- | --- |
| 错因解释、讲评话术、订正引导 | Loop 状态流转 |
| 变式题生成 | 掌握标准（如 2/3 变式对 = 掌握） |
| 家长摘要转写 | **老师确认后才能分发** |
| 讲评优先级建议 | 二次干预触发条件 |

## 8. 产品设计原则（Loop 为一等公民）

### 原则一：每个 Loop 必须有明确的关闭条件

- 学生 Loop：变式通过 → `已掌握`
- 教师 Loop：班级目标错因掌握率 ≥ 阈值 → 本次 Loop 可关闭
- 未关闭 → 自动生成二次干预任务

### 原则二：Loop 必须可观测

三端应能看到的不是孤立功能，而是 **Loop 进度**：

- 老师：本次作业 — 32 人已完成验证，8 人仍薄弱
- 学生：本题 — 订正完成，待变式验证
- 家长：本周 — 主要卡在「图像题理解」，建议配合方式 X

### 原则三：按 Loop 阶段拆模块，而非按功能名堆叠

| 传统拆法（易做散） | Loop 拆法（推荐） |
| --- | --- |
| AI 批改 | Loop 输入层：结构化学习事实 |
| 讲评助手 | Teaching Loop：决策 + 执行 |
| 错题讲解 | Learning Loop：理解 + 订正 |
| 推题 | Learning Loop：验证 |
| 学情报表 | Dual Loop：反馈 + 再诊断 |
| 家长通知 | LoopOutcome：协同视图 |

### 原则四：Agent 是编排者，不是单点按钮

Agent 价值在于：判断 Loop 卡在哪一步 → 生成下一步动作 → 请求老师确认 → 根据反馈进入下一圈或关闭。

## 9. MVP 与 Demo 建议顺序

| 阶段 | 范围 | 证明什么 |
| --- | --- | --- |
| **MVP-1** | 单学生 Learning Loop：错因 → 订正 → 变式 → 掌握 | 「真学会」最直接可感知 |
| **MVP-2** | 加 Teaching Loop 前半段：班级诊断 → 讲评建议 → 老师确认 | 基于真实作业，非拍题工具 |
| **MVP-3** | Dual Loop 咬合：确认后分发 → 掌握回流 → 二次干预名单 | 品类差异，答辩重点 |
| 家长视图 | 轻量只读摘要 | 协同价值，可后置 |

裁剪原则：哪怕只跑「1 份作业 + 1 个班 + 3 个学生」，只要交接点 A、B 真的传通，双端闭环的故事即成立。OCR 可先用电子作答替代，知识点体系先用单章节（如「一次函数」）写死，长期记忆可只存不展示。

## 12. 阶段性结论

**真学会 LearnLoop 的核心不是功能堆叠，而是 Loop 机制：**

1. **Teaching Loop**：让老师的每一次讲评，都有回音。  
2. **Learning Loop**：让学生的每一次订正，都要验证。  
3. **Dual Loop**：让同一份作业数据在老师与学生之间双向流动，直到 Loop 关上。

技术上不需要为了「像 Agent」而做 Multi-Agent；**一个 Loop 引擎、两条工作流、若干 Skill 节点**，足以支撑产品叙事与黑客松 Demo。  
产品设计的重点，是 **把 Loop 的关闭条件、可观测性和数据回流设计清楚** —— 这比架构名词更重要。
